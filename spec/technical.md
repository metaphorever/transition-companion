# Transition Companion — Technical Spec

Current state of the codebase as of migration (2026-05-17). Captures what
exists, not what was originally planned. Open questions and pending audit
findings live in ROADMAP.md, not here.

## Architecture

Single-page React application. Static build, deployed as static files. No
backend. All user state in localStorage. All KB content delivered from a
sibling GitHub repo, cached in localStorage, with a bundled snapshot as
last-resort fallback.

## Data persistence

Two localStorage keys, never mixed:

- `tc_user_data` — everything the user has entered (profile, checklist,
  custom items, recurring items, people). Source of truth for user state.
- `tc_kb_cache` — last-fetched KB content, 24h TTL.

Storage is read on app load, migrated through `mergeWithDefaults` to fill in
missing keys, and written back on every mutation. No partial writes — the
whole `UserData` blob is serialized.

No quota management. If localStorage fills, writes fail silently. See open
question in ROADMAP.md.

## Knowledge base distribution

Three-tier fallback:

1. Valid cached KB in localStorage (≤24h old) — use it.
2. Network fetch from `https://raw.githubusercontent.com/metaphorever/transition-kb/main/`.
3. Stale cache, if any.
4. Bundled snapshot at `public/kb-snapshot/index.json` (last resort).

The KB repo is a separate GitHub repository owned by the same user. The
bundled snapshot is regenerated by a Python script and committed alongside
KB content changes.

## Data model (top level)

Full schema in `src/types/index.ts` (~600 lines). Tree:

```
KBCache:    items · categories · tracks · sequences · jurisdictions · conditions
UserData:   profile · checklist · custom_items · recurring_items · people
```

Key invariants:
- A `ChecklistEntry` is keyed by item slug (KB item or custom item id).
- Items the user has never interacted with have no `ChecklistEntry` — silence
  is not a choice (see D-9 in ROADMAP.md decisions log).
- `intent` is independent of `status`. Active dashboard list filters by
  `intent === 'update'`.
- `Blocker` is a structured first-class object. Resolution does not cascade
  (D-7a).
- Document state is polymorphic (`kind: 'name' | 'marker' | 'full'`)
  capturing the real-world state of physical documents, separate from
  work-progress status.

fallback exists but assumes Python is on PATH. Flag in ROADMAP.md.

## Git workflow

- One worktree per phase. Branch names match worktree names.
- Phase work happens on the worktree branch; main is never directly edited.
- End-of-phase: fast-forward main to the phase branch and push to
  `origin/main`. Standing permission for this — Claude does not ask each
  time, but surfaces the proposed commit message and push target before
  running.
- Force-push to main is never permitted. If FF fails, stop and reconcile
  with Clover.
- Commits never skip hooks (`--no-verify`) or bypass signing unless
  explicitly requested.

## Conventions and standards

- **i18n.** All UI strings live in `src/i18n/en.json`. No hardcoded copy in
  components. Source of truth is the en.json file; other locales are not
  yet present.
- **Tone rules** (see CLAUDE-REFERENCE.md and pre-migration history):
  no first-person plural, no emoji, no "easy wins," no "unfortunately,"
  no sympathy performance.
- **Encoding.** All KB content files are UTF-8. Em-dashes and curly quotes
  written directly, not as cp1252-mangled escapes. `scripts/audit_mojibake.py`
  is a defensive check.
- **Accessibility.** No automated tooling currently — flag in ROADMAP.md.

## Pre-community-release criteria

The app does not need full accessibility audit or hardened privacy posture
during personal dogfood. Both **are** explicit pre-community-release
requirements with their own phases before any public-facing release. The
two have different shapes and demand different treatment.

**Accessibility — additive, "work with this in mind."**

Build with criteria in mind so the eventual audit is a verify-pass rather
than a rework. Concretely:

- Prefer semantic HTML (`<button>`, headings in correct order,
  `<label htmlFor>`).
- Every interactive element must be keyboard-reachable and operable.
- No click handlers on non-button / non-link elements.
- No images without `alt` text.
- No reliance on color alone to convey meaning.

The *design* portion of accessibility — interaction modes, quick-exit /
panic-close patterns, alternative input support, density for low-vision
users — is product design, not an audit-shaped problem. Those questions
get raised in Phase 19 / 20 alongside the dogfood feedback, not deferred
to an "accessibility phase."

**Privacy — prohibitive, hard constraints.**

The privacy posture is already partly encoded — in `spec/product.md`
("the app has no relationship with the user's data"), in the
localStorage-only data model, in the no-first-person-plural copy rule, in
the contribution flow that excludes personal notes. These criteria are
the explicit form of what is already true; new work must not violate them:

- No new network calls beyond the existing KB fetch from
  `raw.githubusercontent.com/metaphorever/transition-kb`.
- No analytics, telemetry, error reporting, or session-replay services.
- No new localStorage / sessionStorage / cookie / IndexedDB keys beyond
  `tc_user_data` and `tc_kb_cache` without a load-bearing justification
  recorded in this file.
- No dependencies that phone home at install or runtime.

The Phase 20 post-dogfood roadmap will lock in the remaining values and
red lines — what the app *promises* to users, what it explicitly refuses
to do even when convenient, how to handle edge cases like debugging,
error diagnosis, and the contribution data flow. Until then: do not
regress, and do not make decisions that would force a rework when those
values are pinned down.

## Dependencies and tradeoffs

Dependencies adopted without an explicit tradeoff write-up at the time
they were introduced. Calling them out now so future decisions don't
treat them as load-bearing without scrutiny:

- **Zustand** vs. Context + useReducer — appropriate for project scale.
  Not revisited.
- **react-router-dom 7** without using the new Data APIs (loaders, actions)
  — fine for a localStorage-only app, but worth knowing if data-fetching
  patterns emerge.
- **Python in the build chain** (KB dep map generator + validator) — adds a
  non-JS toolchain dependency. Could be ported to Node if friction grows.
- **No Prettier, no jsx-a11y, no bundle-size monitor.** Each could be added
  if there's a justification (per the infrastructure rule in product.md).
